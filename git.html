O que é e para que serve?

GIT

GIT

Sistemas de controle de versão são sistemas que gerenciam e registram todas as alterações feitas em um arquivo ou conjunto de arquivos de forma a lhe permitir, futuramente, a recuperação especifica, quando for necessário. Os sistemas de controle de versão permitem que voltemos no tempo, para pegarmos versões especificas e usá-las como base ou até mesmo reverter à versão do software ou sistema que já tenha ou esteja sendo desenvolvido. O Git, por exemplo, é um sistema que foi desenvolvido para controle de versão. Ele é um sistema que tem muitas aplicações e uma delas é a permissão de várias pessoas trabalhando em um mesmo projeto simultaneamente, sem que haja conflito ou bagunça com os arquivos do projeto.

A imagem a seguir exemplifica o funcionamento Git.

O repositório age como um servidor central, ou seja, é onde os dados do projeto e arquivos ficam armazenados. Já as estações estão representando o ambiente onde os programadores estão desenvolvendo o sistema ou software.

Quando a estação de trabalho quer fazer alguma mudança no software/sistema, ela vai ao repositório e pega a versão mais atualizada e traz para o ambiente de trabalho, altera o que for necessário e manda-a de volta para o repositório. Quando essa nova versão é retornada para o repositório, ela é mesclada com o que havia lá e com todos os outros conteúdos que vieram das outras estações formando assim mais uma nova versão.

Como a maioria dos sistemas de controle de versão trabalham de forma centralizada, as alterações realizadas num sistema, serão guardadas no repositório e as estações estarão apenas com uma das versões, a mais atualizada, ou seja, a versão que está sendo atualmente trabalhada na estação.

A principal diferença do Git para os demais sistemas de controle de versionamento é que ele não é um controlador centralizado e sim um sistema distribuído, isso quer dizer que, a cópia de cada versão que está sendo ou foi desenvolvida estará disponível em todas as estações que estiverem trabalhado naquele determinado sistema, permitindo assim, uma velocidade maior por não precisar ficar acessando constantemente o repositório.

Uma vez baixado o projeto do repositório, todas as estações terão disponíveis, todas as versões e alterações que já foram realizadas. É como que cada estação de trabalho fosse um clone do repositório.

A forma como o Git trabalha com projetos é outra vantagem bastante peculiar. Cada projeto é dividido em três estágios representado na figura a baixo.

Primeiro estágio: Working Directory – Diretório de trabalho

Esse diretório é criado quando se vai ao repositório e pega a base do sistema trazendo-a para a estação de trabalho e tudo que está sendo codificado estará dentro do primeiro estágio.

Segundo estágio: Staging Área – Área de teste

A partir do momento em que se der um Git add, iniciamos o segundo estágio. Nele, os arquivos que estavam sendo editados, vão para um local semelhante ao de uma sala de espera, também, chamada de área de teste. Esta área é o local onde tudo que foi alterado fica antes de ser enviado de volta para o repositório.

Terceiro estágio: Staging Área – Área de teste

Depois que se der o comando Git commit, os arquivos vão para o repositório final, o Git Directory, que é a base de onde foram pegos.

É justamente essa divisão que torna o Git uma ferramenta diferenciada, confiável e flexível se comparada às demais ferramentas de controle de versionamento resultando em um trabalho mais simples e rápido.
