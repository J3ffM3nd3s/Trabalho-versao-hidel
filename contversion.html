SISTEMA DE CONTROLE DE VERSIONAMENTO

Os sistemas de versionamentos ou sistema de controle de versão surgiram há bastante tempo, em meados da década de 70, porém, só depois de uma década, surgiu um que formou a base de arquitetura que conhecemos nos dias atuais. Anos depois vem a evolução tornando praticável o uso desses sistemas.

O QUE É E PARA QUE SERVE?

Composto por duas partes, sendo elas: estação de trabalho e repositório, os sistemas de controle de versão foram desenvolvidos para gerenciar todas as alterações feitas em um software ou sistema que já tenha sido ou esteja sendo desenvolvido.

O repositório armazena o histórico evolutivo do projeto e registra todas as alterações realizadas;

Estação de trabalho contém a cópia dos arquivos do projeto e que é monitorada para identificar as mudanças realizadas. Essa área é individual e isolada das demais áreas de trabalho.

O Controle de versão apoia o desenvolvimento das seguintes formas:

Registro do Histórico: Registra toda a evolução do projeto, cada alteração sobre cada arquivo. Com essas informações sabe-se quem fez o que, quando e onde. Além disso, permite reconstruir uma revisão específica do arquivo sempre que desejado;

Colaboração Concorrente: O controle de versão possibilita que vários desenvolvedores trabalhem em paralelo sobre os mesmos arquivos sem que um sobrescreva o código de outro, o que traria reaparecimento de defeitos e perda de funcionalidades;

Variações no Projeto. Mantém linhas diferentes de evolução do mesmo projeto. Por exemplo, mantendo uma versão 1.0 enquanto a equipe prepara uma versão 2.0.

Podendo ser classificados em dois tipos: Centralizado e Distribuído.

DIFERENÇA ENTRE SISTEMA CENTRALIZADO E SISTEMA DISTRIBUÍDO

Ambos possuem repositórios estação de trabalho, mas, a principal diferença entre eles está em como cada um é arranjado.

SISTEMA CENTRALIZADO

São sistemas de fácil aprendizado em relação a outros sistemas e eles trabalham apenas com um servidor central e diversas estações de trabalho uma para cada desenvolvedor. É a partir da estação que os desenvolvedores conseguem visualizar e compartilhar seu trabalho com os colaboradores, gerando um fluxo de dados, que obrigatoriamente, precisa passar pelo repositório.

Os sistemas centralizados oferecem muitas vantagens como: controle central sobre as contribuições que foram realizadas, o responsável consegue fazer uma administração mais facilmente controlada das atualizações que foram feitas por cada desenvolvedor que faz parte do projeto.

SISTEMA DISTRIBUIDO

Nos sistemas distribuídos, os desenvolvedores do projeto fazem copias completas do repositório e não ficam dependentes remotamente do mesmo. O maior benefício desse tipo de sistema é que qualquer um dos colaboradores poderá ter uma cópia completa do repositório ou da ramificação do projeto na sua estação de trabalho. Dessa forma a redundância do sistema é aumentado e por consequência acaba eliminando o ponto de falha, pois, ele deixa de ser centralizado e acaba se tornando flexível. Essa característica não será encontrada nos sistemas centralizados.

Podemos destacar também, nos sistemas distribuídos, a comunicação entre si das estações de trabalho, podendo usar um servidor remoto para centralizar o fluxo de dados.

Por serem flexíveis, os sistemas distribuídos são mais complexos, porém, mais vantajosos e são mais interessantes para as equipes que trabalham simultaneamente no mesmo projeto com grandes grupos de pessoas.

Os dois tipos de sistema de versão além de extremamente úteis se destacam se compararmos com sistemas de controle manual de projeto. O sistema centralizado dispõe de um servidor central e varias estações de trabalho e forma uma topologia de rede estrela. O sistema distribuído permite múltiplos repositórios locais ao invés de um repositório remoto centralizado e o acesso a rede só é necessário quando houver a publicação das alterações ou quando for preciso verificar as mudanças que foram realizadas pelos demais colaboradores do projeto.
